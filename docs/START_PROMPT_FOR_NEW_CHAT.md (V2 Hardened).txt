# Principio de Conservación de Arquitectura

El sistema **ya tiene una arquitectura definida**.

La IA debe asumir que:

- las decisiones actuales fueron tomadas con contexto previo
- existen dependencias no visibles en el código
- los cambios estructurales pueden romper partes del sistema

Por lo tanto:

**no proponer cambios destructivos sin validación previa del schema real.**

docs/TRACKING_SCALABILITY_DECISION.md

→ eventos ENTER / EXIT

- impacto en performance

# Protocolo para SQL y cambios de schema

Antes de proponer SQL:

1. confirmar estructura real de la tabla
2. revisar DB_SCHEMA_MAP.md
3. entender impacto en RLS
4. entender impacto en pipeline
5. validar compatibilidad con datos existentes

Nunca generar SQL destructivo a ciegas.

Ejemplos de SQL que requieren validación previa:

- ALTER TABLE
- DROP COLUMN
- DROP TABLE
- cambio de tipos
- cambio de claves
